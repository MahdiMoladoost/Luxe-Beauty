# Luxe Beauty

پلتفرم فارسی و راست‌به‌چپ برای جستجو، مقایسه و رزرو آنلاین سالن‌ها و خدمات زیبایی، همراه با پنل مشتری، سالن، آرایشگر، ادمین و پشتیبانی.

## وضعیت فعلی

این شاخه، نمونه اولیه رابط کاربری را به یک MVP اجرایی متصل تبدیل می‌کند:

- صفحه اصلی، جستجوی سالن و پروفایل سالن
- رزرو چندمرحله‌ای از انتخاب خدمت تا رسید
- انتخاب آرایشگر سازگار با چند خدمت
- Availability API و Slot Hold هفت‌دقیقه‌ای
- کنترل تداخل هنگام Hold و ثبت نهایی
- OTP توسعه‌ای با انقضا، محدودیت ارسال و سقف تلاش
- محاسبه قیمت و بیعانه در سمت سرور
- پنل‌های مشتری، سالن، آرایشگر و ادمین نمایشی/عملیاتی
- ثبت سالن، تماس و تیکت پشتیبانی با کد پیگیری
- صفحات دسته‌بندی، تخفیف، مجله، درباره، FAQ، قوانین و حریم خصوصی
- صفحات 404، خطای پرداخت و تعمیرات
- طرح PostgreSQL تولیدی در `database/schema.sql`

## کد OTP در توسعه

در محیط غیرتولیدی، پاسخ API کد OTP را با فیلد `demoCode` برمی‌گرداند تا جریان رزرو بدون سرویس پیامک تست شود. در `NODE_ENV=production` این مقدار هرگز بازگردانده نمی‌شود.

## ذخیره‌سازی فعلی MVP

APIهای اجرایی این شاخه برای تست محلی از حافظه پردازش Node.js استفاده می‌کنند. این داده با Restart، Scale-out یا جابه‌جایی Function پایدار نمی‌ماند. برای انتشار واقعی باید Repository لایه سرور به PostgreSQL و طرح `database/schema.sql` متصل شود.

این محدودیت عمداً در `/api/health` نیز نمایش داده می‌شود تا محیط آزمایشی با تولید اشتباه نشود.

## متغیرهای محیطی تولید

نمونه نام متغیرها در `.env.example` قرار دارد. هیچ توکن، رمز یا کلید خصوصی نباید در Git ثبت شود.

حداقل اتصال‌های خارجی برای تولید:

- PostgreSQL
- سرویس پیامک OTP و اعلان
- درگاه پرداخت و Callback امن
- Object Storage برای تصاویر و مدارک
- سرویس نقشه
- ایمیل تراکنشی، اختیاری
- Error tracking و observability

## قواعد حیاتی رزرو

1. زمان در دیتابیس UTC و در رابط با منطقه زمانی ایران نمایش داده شود.
2. قیمت نهایی فقط در سرور محاسبه شود.
3. هنگام انتخاب زمان، Slot Hold با زمان انقضا ساخته شود.
4. ثبت نهایی داخل Transaction و با بررسی دوباره Availability انجام شود.
5. پرداخت، Callback و ثبت رزرو باید Idempotent باشند.
6. تغییر قیمت، وضعیت رزرو، پرداخت، دسترسی و Impersonation باید Audit Log داشته باشد.
7. رزروهای فعال یک آرایشگر نباید در دیتابیس هم‌پوشانی داشته باشند.

## امنیت

- OTP در دیتابیس باید Hash شود.
- Session باید HttpOnly، Secure و SameSite باشد.
- Rate limit برای OTP، ورود، جستجو و فرم‌های عمومی لازم است.
- فایل‌های مدارک خصوصی‌اند و فقط با URL امضاشده نمایش داده می‌شوند.
- دسترسی هر سالن باید به `salon_id` خودش محدود باشد.
- Impersonation ادمین نیازمند دلیل، زمان انقضا و Audit Log کامل است.
- اسرار فقط در Secret Manager یا متغیر محیطی نگهداری شوند.
