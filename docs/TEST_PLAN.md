# برنامه تست و معیار انتشار

## Gateهای اجباری Pull Request

- نصب وابستگی با `npm ci`
- TypeScript بدون خطا با `npx tsc --noEmit`
- Build تولید با `npm run build`
- بازبینی تغییرات امنیتی و مالی
- عدم وجود Secret در Git
- عدم تغییر مستقیم `main`

CI این شاخه TypeScript و Build را روی هر Pull Request به `main` اجرا می‌کند.

## تست موتور رزرو

### محاسبه قیمت

- قیمت عادی
- قیمت تخفیف کمپین
- کد تخفیف معتبر و نامعتبر
- جلوگیری از مبلغ منفی
- بیعانه بیشتر از مبلغ کل
- خدمات متعلق به سالن دیگر
- شناسه خدمت تکراری

### سازگاری پرسنل

- یک خدمت با چند آرایشگر
- چند خدمت با آرایشگر مشترک
- چند خدمت بدون آرایشگر مشترک
- آرایشگر غیرفعال
- قیمت یا مدت اختصاصی پرسنل

### زمان‌بندی

- هم‌پوشانی در ابتدا، انتها و داخل بازه
- دو رزرو چسبیده بدون هم‌پوشانی
- زمان استراحت
- تعطیلی سالن
- مرخصی پرسنل
- عبور خدمت از ساعت پایان شیفت
- تغییر ساعت رسمی و منطقه زمانی

## تست Hold

- ایجاد Hold موفق
- رد Hold روی رزرو فعال
- رد Hold روی Hold فعال دیگر
- آزاد شدن بعد از هفت دقیقه
- تعویض Hold قبلی همان مشتری
- ثبت نهایی قبل و بعد از انقضا
- دو درخواست هم‌زمان برای یک اسلات
- Worker غیرفعال‌سازی Hold منقضی

انتظار تولید: دقیقاً یکی از دو درخواست هم‌زمان موفق شود.

## تست OTP و Session

- شماره صحیح و غلط
- تبدیل ارقام فارسی و عربی
- محدودیت ارسال مجدد
- کد منقضی
- کد اشتباه و سقف تلاش
- Replay کد مصرف‌شده
- Cookie امن و HttpOnly
- Rate limit بر اساس شماره، IP و Device fingerprint
- خروج و لغو Session

## تست پرداخت

- ایجاد پرداخت با Idempotency Key
- Callback موفق
- Callback ناموفق
- Callback تکراری
- مبلغ Callback متفاوت
- رزرو منقضی پیش از Callback
- Refund کامل و جزئی
- قطعی Gateway و Retry
- تطبیق روزانه تراکنش‌ها

انتظار تولید: هیچ Callback تکراری نباید پرداخت، رزرو یا Ledger را دوباره ثبت کند.

## تست دسترسی نقش‌ها

- مشتری فقط رزروهای خودش را می‌بیند.
- پرسنل فقط برنامه و مشتریان مجاز سالن خودش را می‌بیند.
- مدیر سالن به سالن دیگر دسترسی ندارد.
- پشتیبانی به داده مالی حساس بیش از نیاز دسترسی ندارد.
- ادمین Impersonation را با دلیل و Audit Log انجام می‌دهد.
- حساب تعلیق‌شده عملیات نوشتن انجام نمی‌دهد.

## تست سالن و خدمات

- ثبت درخواست تکراری
- تایید و رد مدارک
- انتشار فقط پس از تایید
- تغییر ساعات کاری با رزرو آینده
- غیرفعال کردن خدمت دارای رزرو آینده
- تغییر قیمت و Snapshot رزرو قبلی
- حذف پرسنل دارای رزرو آینده

## تست اعلان

- تایید رزرو
- یادآوری قبل از نوبت
- لغو و جابه‌جایی
- پرداخت و Refund
- نظر جدید
- درخواست تسویه
- Retry بدون ارسال تکراری
- لغو پیام تبلیغاتی بدون غیرفعال شدن پیام تراکنشی

## تست UI و دسترس‌پذیری

- RTL کامل در موبایل، تبلت و دسکتاپ
- پیمایش با کیبورد
- Focus واضح
- Label برای ورودی‌ها
- کنتراست متن و دکمه
- Screen reader برای وضعیت خطا و Loading
- کاهش حرکت برای کاربرانی با `prefers-reduced-motion`
- Empty state و Error state تمام فهرست‌ها
- قطع شبکه وسط رزرو
- Refresh صفحه در هر مرحله

## تست صفحات کلیدی E2E

1. جستجو → پروفایل سالن → چند خدمت → آرایشگر → زمان → OTP → Hold → پرداخت → رسید
2. اسلات هم‌زمان توسط دو کاربر
3. انقضای Hold در صفحه مرور
4. پرداخت ناموفق و تلاش مجدد
5. لغو و Refund مطابق سیاست
6. ثبت سالن و پیگیری وضعیت
7. تیکت پرداخت با اولویت بالا
8. ثبت نظر فقط برای رزرو تکمیل‌شده

## Performance

اهداف اولیه پیشنهادی، پس از اندازه‌گیری واقعی قابل تنظیم‌اند:

- P95 پاسخ API خواندنی کمتر از ۵۰۰ms
- P95 Availability کمتر از ۸۰۰ms
- نرخ خطای API کمتر از ۱٪
- LCP صفحات عمومی کمتر از ۲.۵ ثانیه روی شبکه موبایل مناسب
- Queue lag اعلان‌های تراکنشی کمتر از ۳۰ ثانیه

## امنیت پیش از انتشار

- Dependency audit
- SAST و Secret scanning
- بررسی OWASP ASVS بخش‌های احراز هویت، Session، دسترسی و آپلود
- تست SQL injection و XSS

- Backup restore drill
- Runbook رخداد امنیتی

## معیار Go/No-Go

انتشار فقط در صورتی مجاز است که:

- تمام Gateهای CI سبز باشند.
- هیچ خطای Critical یا High باز در رزرو، پرداخت، دسترسی و امنیت وجود نداشته باشد.
- درگاه و پیامک در Sandbox و سپس Production تایید شده باشند.
- سیاست‌های حقوقی و بازگشت وجه تصویب شده باشند.
- Monitoring، Backup و Runbook آماده باشند.
- یک تست هم‌زمانی واقعی روی دیتابیس تولیدمانند عبور کرده باشد.
