# معماری پیشنهادی لوکس بیوتی

## ۱. مرزهای سیستم

محصول به پنج سطح دسترسی اصلی تقسیم می‌شود:

- مشتری
- مدیر سالن
- آرایشگر / پرسنل
- پشتیبانی
- ادمین پلتفرم

هر درخواست سرور باید علاوه بر احراز هویت، نقش و محدوده مالکیت داده را کنترل کند. مدیر یا پرسنل یک سالن نباید با تغییر شناسه URL به اطلاعات سالن دیگری دسترسی پیدا کند.

## ۲. اجزای اصلی

### Web Application

Next.js App Router رابط عمومی و پنل‌ها را ارائه می‌کند. صفحات عمومی قابل ایندکس هستند؛ صفحات حساب و عملیات مالی باید خصوصی و بدون Cache عمومی باشند.

### Application API

API مسئول اعتبارسنجی، مجوز، محاسبه قیمت، رزرو، پرداخت، اعلان و ثبت Audit Log است. فرانت‌اند هیچ‌گاه منبع حقیقت قیمت یا Availability نیست.

### PostgreSQL

داده‌های تراکنشی، مالی، رزرو، دسترسی و Audit Log در PostgreSQL نگهداری می‌شوند. طرح اولیه در `database/schema.sql` قرار دارد.

### Queue and Workers

کارهای غیرهم‌زمان باید از صف اجرا شوند:

- ارسال OTP و اعلان
- یادآوری نوبت
- آزاد کردن Hold منقضی
- تطبیق وضعیت پرداخت
- تولید گزارش
- پردازش تصویر
- تسویه مالی

### External Integrations

- درگاه پرداخت
- پیامک
- فضای ذخیره فایل و CDN
- نقشه و Geocoding
- ایمیل تراکنشی، اختیاری
- Error tracking و Log aggregation

## ۳. جریان رزرو

1. کاربر سالن و خدمات را انتخاب می‌کند.
2. سرور سازگاری خدمات و آرایشگر را بررسی می‌کند.
3. Availability از ساعات کاری، استثناها، Holdها و رزروهای فعال محاسبه می‌شود.
4. سرور قیمت، مدت و بیعانه را محاسبه می‌کند.
5. یک Slot Hold فعال با انقضای هفت‌دقیقه‌ای داخل Transaction ثبت می‌شود.
6. کاربر OTP را تایید و در صورت نیاز وارد درگاه می‌شود.
7. Callback درگاه با Idempotency Key تایید می‌شود.
8. در یک Transaction نهایی، Hold و Availability دوباره بررسی و Booking ایجاد می‌شود.
9. Hold غیرفعال، Ledger ثبت و اعلان‌ها در Outbox/Queue قرار می‌گیرند.

## ۴. جلوگیری از Double Booking

کنترل فقط در حافظه یا فرانت‌اند کافی نیست. در PostgreSQL برای رزروهای فعال یک Exclusion Constraint روی بازه زمانی هر `staff_id` در نظر گرفته شده است.

برای Hold نیز فیلد `active` وجود دارد. Worker باید Holdهای منقضی را غیرفعال کند و هر Query علاوه بر `active`، شرط `expires_at > now()` را بررسی کند. ساخت Hold و ثبت نهایی باید Transactional باشد.

## ۵. قیمت و پول

- مبلغ‌ها به‌صورت عدد صحیح و بر حسب تومان ذخیره می‌شوند.
- محاسبه قیمت در سرور انجام می‌شود.
- نام خدمت، قیمت، مدت و قوانین هنگام رزرو Snapshot می‌شوند.
- هر تغییر قیمت دارای نسخه و Audit Log است.
- پرداخت و Callback باید Idempotent باشند.
- موجودی کیف پول از Ledger مشتق می‌شود و تغییر مستقیم بدون Entry مجاز نیست.

## ۶. زمان و تقویم

- ذخیره در دیتابیس: UTC با `timestamptz`
- منطقه تجاری: `Asia/Tehran`
- نمایش رابط: شمسی
- تبدیل تقویم فقط در لایه نمایش انجام می‌شود.
- قواعد ساعات کاری با منطقه زمانی سالن محاسبه می‌شوند.

نسخه فعلی رابط برای ورودی آزمایشی از Date input میلادی استفاده می‌کند. جایگزینی آن با Datepicker شمسی دارای تست تبدیل، بخشی از آماده‌سازی تولید است.

## ۷. احراز هویت و امنیت

- OTP باید Hash و دارای انقضا، Rate limit و سقف تلاش باشد.
- Session در Cookie از نوع HttpOnly، Secure و SameSite ذخیره شود.
- عملیات حساس نیازمند Re-authentication هستند.
- مدارک سالن خصوصی و دارای Signed URL کوتاه‌عمر هستند.
- اسرار فقط در Secret Manager نگهداری می‌شوند.
- ورودی‌ها با Schema مشترک اعتبارسنجی می‌شوند.
- رخدادهای امنیتی، تغییر دسترسی و Impersonation ثبت می‌شوند.

## ۸. Audit Log

حداقل رخدادهای زیر ثبت می‌شوند:

- ایجاد، تایید، لغو و جابه‌جایی رزرو
- تغییر قیمت و قوانین سالن
- پرداخت، Refund و Settlement
- تغییر نقش و دسترسی پرسنل
- بررسی مدارک سالن
- Moderation نظر
- ورود ادمین به پنل سالن

Audit Log نباید توسط کاربران عملیاتی قابل ویرایش باشد و برای داده حساس باید Redaction داشته باشد.

## ۹. اعلان

اعلان‌ها از طریق جدول `notifications` و Worker ارسال می‌شوند. هر پیام `idempotency_key` دارد تا Retry باعث ارسال تکراری نشود.

کانال‌ها:

- In-app
- SMS
- Email
- WhatsApp در صورت قرارداد و امکان فنی

قالب، زبان، وضعیت ارسال، Retry و خطا باید ثبت شوند.

## ۱۰. جستجو و مکان

MVP می‌تواند با Queryهای PostgreSQL شروع شود. با رشد داده، جستجوی متن، Autocomplete و فاصله جغرافیایی می‌توانند به PostgreSQL Full Text/PostGIS یا موتور جستجوی اختصاصی منتقل شوند.

## ۱۱. Observability

برای محیط تولید لازم است:

- Request ID مشترک بین API، Worker و Gateway
- Structured logging بدون داده حساس
- Error tracking
- متریک نرخ رزرو موفق، خطای پرداخت، زمان پاسخ و Queue lag
- Alert برای افزایش Double-booking conflict، Callback نامعتبر و شکست اعلان

## ۱۲. وضعیت پیاده‌سازی این شاخه

این شاخه یک MVP قابل مرور با Store حافظه‌ای ارائه می‌کند تا جریان رزرو و قرارداد API قابل آزمایش باشد. Store حافظه‌ای برای محیط چندنمونه‌ای یا تولید مناسب نیست. اتصال Repositoryها به PostgreSQL، درگاه، پیامک و فضای ذخیره نیازمند سرویس و تنظیمات واقعی است و نباید با داده یا کلید ساختگی به‌عنوان تولید معرفی شود.
